# 实验日志

## 文档目的
本文件作为自动小说系统的统一实验入口，用于记录每一次结构试跑、规则验证、工作流验证、样例验证与失败复盘。

它回答的不是“系统理论上能做什么”，而是：

- 这次具体试了什么
- 输入是什么
- 过程怎么跑的
- 哪一步卡住了
- 哪些规则有效，哪些规则失效
- 哪些字段够用，哪些字段不够用
- 下一步该保留什么、修改什么、删除什么

它不是正式设计文档，而是把每一次试跑留下可复盘、可比较、可积累的证据。

---

## 1. 为什么必须有实验日志

到目前为止，你已经完成了：

- 项目层
- 概念层
- 数据模型层
- 规则层
- 工作流层
- 示例层

这些内容说明系统在结构上已经成立。接下来更重要的问题会变成：

- 运行时哪里真的顺
- 哪里只是文档上看起来顺
- 哪个字段经常用不到
- 哪个规则在样例里抓不到问题
- 哪条工作流会不会中途断掉
- 同一个问题会不会反复出现

如果没有实验日志，后面很容易陷入“感觉这里改过”“好像之前试过”“忘了上次为什么这么定”的循环。

所以实验日志的目标，是把“感觉”变成“记录”。

---

## 2. 适用范围

本日志适用于记录以下类型的实验：

### 2.1 对象实验

例如：

- `CharacterModel` 是否足以支撑角色审查
- `NarrativeState` 是否能准确表达当前局势
- `FactLedger` 粒度是否合适

### 2.2 规则实验

例如：

- `world_legality_rules` 能不能抓到越界
- `information_release_rules` 会不会过宽或过严
- `failure_types` 是否足够稳定

### 2.3 工作流实验

例如：

- `Rebuild Workflow` 是否能从断更小说重建出当前状态
- `Continue Workflow` 是否会自然生成有效 `PlotUnit`
- `Rewrite Workflow` 能不能做到最小修复

### 2.4 样例实验

例如：

- `mock_project_case` 能否连续跑 3 到 5 个单元
- 当前样例是否太简单，导致规则看起来总是成立

### 2.5 失败复盘实验

例如：

- 某次续写为什么变成人设漂移
- 某次审查为什么漏掉明显问题
- 某次改写为什么修一个洞开三个洞

---

## 3. 核心原则

### 3.1 一次实验只回答一个主问题

不要一次实验同时想验证十件事。

坏实验：

- 同时验证 schema、规则、工作流、样例、风格、节奏

好实验：

- 只验证当前 `Continue Workflow` 是否会优先兑现近程后果

### 3.2 记录输入，而不是只记录结论

很多实验失败并不是因为系统错，而是因为输入不完整。

所以每次实验至少要记录：

- 用了哪些对象
- 用了哪个样例
- 输入是否完整
- 哪些地方是手工补的

### 3.3 记录失败比记录成功更重要

成功常常只说明“这次刚好跑通了”。失败更能暴露：

- 哪个边界不清
- 哪个字段不够
- 哪条规则没有落点
- 哪个工作流顺序有问题

### 3.4 结论必须能导向动作

实验结论不要只写“感觉一般”“还行”“有点怪”，而应尽量落成：

- `keep`
- `patch`
- `rewrite`
- `replan`
- `defer`
- `watch`
- `drop`

---

## 4. 日志结构建议

建议每一条实验记录尽量包含以下 8 个部分：

1. 实验编号
2. 实验目标
3. 输入材料
4. 操作过程
5. 实验观察
6. 结果判定
7. 问题与原因
8. 后续动作

---

## 5. 实验编号规则

建议统一使用：

```text
EXP-YYYYMMDD-序号
```

例如：

- `EXP-20260326-01`
- `EXP-20260326-02`

如果以后要分类型，也可以扩展成：

```text
EXP-WF-20260326-01
EXP-RULE-20260326-02
EXP-EXAMPLE-20260326-03
```

其中：

- `WF` = workflow
- `RULE` = rules
- `MODEL` = data model
- `EXAMPLE` = sample / mock case
- `REVIEW` = review pipeline

---

## 6. 实验结果分级

建议统一把实验结果分成四档：

- `成立`：本次实验目标基本达成，当前方案可继续保留
- `基本成立`：主目标达成，但存在局部问题，需要小修
- `不稳定`：有时成立、有时不成立，或在边界场景下明显失真
- `失败`：当前方案不能支撑目标，必须修改或重做

---

## 7. 后续动作标签

为避免结论太虚，建议每次实验最后都落一个动作标签：

- `keep`
- `patch`
- `rewrite`
- `replan`
- `defer`
- `watch`
- `drop`

## 10. 日志模板

下面给出推荐模板。

```md id="exp00template"
## EXP-YYYYMMDD-01

### 实验标题
一句话描述本次实验。

### 实验类型
- workflow / rule / model / example / review / rewrite

### 实验目标
本次实验只验证什么问题？

### 输入材料
- 使用了哪些文件
- 使用了哪些对象
- 是否基于 mock case
- 是否有人工补充输入

### 操作过程
按顺序写：
1. 做了什么
2. 调用了什么
3. 观察了什么

### 实验观察
记录：
- 哪一步顺
- 哪一步卡
- 哪一步需要人工脑补
- 哪一步结果不稳定

### 结果判定
- 成立 / 基本成立 / 不稳定 / 失败

### 发现的问题
列出本次发现的具体问题。

### 原因判断
对问题给出当前最可能原因。

### 后续动作
- keep / patch / rewrite / replan / defer / watch / drop

### 备注
补充说明。
```

## 11. 首条示例实验记录

```md id="exp00sample1"
## EXP-20260326-01

### 实验标题
验证 Continue Workflow 是否会优先兑现 scene_014 已建立的近程后果

### 实验类型
workflow

### 实验目标
验证当前 `04_workflows/03_continue_workflow.md` 在接续 `pu_scene_014 / 雨夜夺令` 后，是否会优先推进以下内容之一：

- 旧伤后果
- 巡查追踪压力
- c002 保密立场

而不是直接跳去无关新支线。

### 输入材料
- `05_examples/01_sample_workspec.md`
- `05_examples/02_sample_worldmodel.md`
- `05_examples/03_sample_characters.md`
- `05_examples/04_sample_plotunit.md`
- `05_examples/05_sample_factledger.md`
- `05_examples/07_mock_project_case.md`

### 操作过程
1. 读取当前 `NarrativeState` 的压缩快照
2. 提取当前最紧迫风险：
   - 旧伤
   - 巡查逼近
   - 保密压力
3. 对照 `Continue Workflow` 中的优先级原则
4. 检查下一步生成方向是否自然指向“后果兑现推进”

### 实验观察
- 当前 workflow 文档在原则层已经明确“近程后果优先”
- 该原则能和 mock case 当前状态自然对齐
- 但目前还没有第二个真正生成出来的 `PlotUnit` 样例，因此验证仍停留在“结构级验证”
- 说明工作流顺序成立，但还未进入“运行级验证”

### 结果判定
基本成立

### 发现的问题
1. `Continue Workflow` 目前更像规则化说明，还没有附一个真实的 `pu_scene_015` 样例
2. “优先兑现后果”虽然被写明，但缺少一个失败对照样例
3. 当前无法验证：如果系统错误地跳去新支线，审查是否一定能拦住

### 原因判断
- 示例层目前只跑到了 `pu_scene_014`
- 缺少连续第二单元样例，导致 workflow 仍偏静态验证

### 后续动作
patch

### 备注
下一步建议补：
- `pu_scene_015` 的样例
- 或先在 experiments 中补一条“错误续写对照实验”
```

- 当前示例层主要是正例

## 13. 当前阶段最重要的实验目标

如果只选一个最重要目标，建议当前阶段先围绕这件事记录实验：

**验证这套系统在 mock case 上能不能连续跑两轮，而不只是跑一轮。**

因为现在你已经证明了：

- 一轮对象建模可以成立
- 一轮 `PlotUnit` 可以成立
- 一轮审查可以成立

接下来真正关键的是：

- 第二轮续写会不会漂
- 第二轮审查会不会漏
- 第二轮改写会不会失控

这会比继续堆抽象文档更有价值。

## 14. 一句话总结

`00_experiment_log.md` 的作用，是把这套系统从“文档已经很多”推进到“真的开始试跑”，并且让每一次试跑都留下可比较、可复盘、可转化为设计修改的实验记录。
