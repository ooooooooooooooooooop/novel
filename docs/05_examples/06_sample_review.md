# 示例：Review

## 文档目的
本文件提供一个正式审查样例，用于验证：

- `Review Workflow` 是否真的能落地执行
- 一次审查是否能从对象层出发，而不是停留在模糊评价
- 审查结果是否能稳定映射为 `PlotUnit` 审查字段、`ReviewIssue` 与后续动作建议
- 审查是否能形成真正的“发现问题 → 定位问题 → 建单 → 决定放行/返工”的闭环

本文件对应：

- `01_concepts/08_reviewissue.md`
- `02_data_models/08_reviewissue_schema.md`
- `03_rules/07_review_rules.md`
- `03_rules/08_failure_types.md`
- `04_workflows/02_review_workflow.md`

---

## 1. 示例定位

这份样例继续沿用当前 mock project 的统一案例：

- 作品：`残雪归宗`
- 世界：`北衡界`
- 当前关键推进单元：`pu_scene_014 / 雨夜夺令`
- 当前相关状态：输入状态 `s_014_scene`，输出状态 `s_015_scene`
- 当前相关角色：`c001 / 沈青`、`c002 / 顾临渊`、`c003 / 裴照`
- 当前关键事实：`f_014` 令牌归属变化、`f_015` `c002` 已确认异常痕迹、`f_016` 旧伤加重、`f_017` 巡查风险激活
- 当前关键承诺：`fg_001` 身世 / 残魂主线，`fg_002` `c002` 是否会保密

这份文件不追求“找很多毛病”，而追求展示一次健康的正式审查是什么样。

---

## 2. 使用场景

这份样例主要用于验证以下问题：

### 2.1 审查是不是从对象出发

系统能不能明确读取：

- 当前 `PlotUnit`
- 当前前后状态
- 当前角色模型
- 当前事实账本
- 当前承诺线程

而不是只说“感觉不错”或“感觉有点怪”。

### 2.2 审查是不是能给出清晰结论

系统能不能说清：

- 哪些地方成立
- 哪些地方只是有警告
- 哪些问题已经需要建单
- 当前到底能不能放行

### 2.3 审查是不是能服务后续动作

系统能不能在审查后明确决定：

- 继续推进
- 带警告推进
- 进入改写
- 进入重规划

---

## 3. 本次审查对象

当前审查对象为：

- `PlotUnit`：`pu_scene_014`
- 标题：`雨夜夺令`

本次审查目标是检查这次推进是否同时满足以下六项：

1. 状态转移成立
2. 世界合法性成立
3. 角色一致性成立
4. 时间与信息顺序成立
5. 承诺推进成立
6. 是否需要生成正式 `ReviewIssue`

---

## 4. 审查输入摘要

### 4.1 PlotUnit

当前单元已知：

- `unit_goal`：在巡查队到来前决定令牌归属，并逼出关系与风险走向
- `initiator`：`c001`
- `key_decision`：不再等待 `c002` 站队，先手夺令
- `progression_strength`：预期为 `strong`
- `legality_status`：预期为 `valid_with_cost`
- `character_consistency_status`：预期为 `valid_with_cost`

### 4.2 输入状态

- 时间：试炼开始前夜
- 地点：黑水泽外围废弃驿亭
- 当前目标：在不暴露身份的前提下拿到令牌
- 当前冲突：令牌争夺、信任危机、身份暴露风险

### 4.3 输出状态

- 主目标已转为带令撤离并控制秘密扩散
- 关系张力升级
- 风险上升
- `c002` 的知情状态发生变化

### 4.4 相关角色

- `c001`：高压下倾向独扛，怕依赖别人
- `c002`：倾向观察和延迟决断，但对 `c001` 有保护冲动
- `c003`：擅长试探、误导和施压

### 4.5 相关事实

- `f_014`：令牌当前归 `c001` 所有
- `f_015`：`c002` 已确认异常痕迹
- `f_016`：`c001` 旧伤加重
- `f_017`：巡查队重新锁定概率上升
- `f_018`：`c001` 与 `c002` 进入高风险半合作边缘
- `f_019`：局势主目标已转为撤离与封口

### 4.6 相关承诺线程

- `fg_001`：主角身世与残魂主线
- `fg_002`：`c002` 是否会替 `c001` 保密

## 5. 审查过程示例

### 5.1 输入完整性检查

检查项目如下：

- 是否有当前 `PlotUnit`
- 是否有输入状态
- 是否有输出状态
- 是否有相关角色模型
- 是否有相关事实账本
- 是否有相关承诺线程

结果是全部具备，因此可以进行高置信正式审查。

### 5.2 状态转移审查

调用依据：

- `03_rules/01_state_transition_rules.md`

本次核查重点是：

1. 是否有明确输入状态
2. 是否有关键行动者
3. 是否有关键决策
4. 是否有阻力与约束
5. 是否有输出状态
6. 是否有明确状态变化

当前判断如下：

- 输入状态成立
- `initiator = c001`，关键行动者成立
- `c001` 决定先手夺令，关键决策成立
- 阻力充分存在：暴雨、巡查队、旧伤、`c002` 立场不明、`c003` 持续施压
- 输出状态成立
- 状态变化至少覆盖资源、信息、目标、风险与冲突五层

结论：

- `progression_strength = strong`
- 状态转移成立
- 不构成 `weak_progression`

### 5.3 世界合法性审查

调用依据：

- `03_rules/02_world_legality_rules.md`

核查重点：

1. 是否违反硬规则
2. 是否跳过应有代价
3. 是否触发合理世界后果
4. 是否无视制度、资源或环境约束

当前判断如下：

- 没有直接违反硬规则
- 已明确建立代价：`f_016` 旧伤加重、异常灵压外泄提高追踪风险
- 制度与世界后果已建立但尚未完全兑现：`f_017` 巡查队重新锁定概率上升
- 资源合法性成立：令牌归属变化已入账，令牌仍是稀缺资源，不是被随意复制

结论：

- `legality_status = valid_with_cost`
- 当前不生成 `world_violation`
- 但后续 1 到 2 个单元若不兑现追踪 / 旧伤后果，应升级为 `missing_cost` 或 `missing_consequence`

### 5.4 时间合法性审查

调用依据：

- `03_rules/04_timeline_rules.md`

当前判断：

- 事件顺序成立：先争夺，再逼近暴露，再切换目标到撤离与封口
- 因果顺序成立：先使用高压能力，再出现旧伤与异常灵压后果
- 信息顺序成立：`c002` 是在现场观察后才获得确认
- 当前没有直接并行冲突

结论：

- 时间合法性成立
- 不构成 `timeline_error`

### 5.5 信息释放审查

调用依据：

- `03_rules/05_information_release_rules.md`

当前判断：

- 知情权限成立：`c002` 知道异常痕迹，`c003` 尚未掌握完整本质，监察司仍未知，读者 / 系统已知追踪风险被激活
- 揭露时机成立：没有直接揭晓残魂真相，只推进到“半暴露”阶段
- 揭露粒度合理：给了局部证据，没有一次性解释全部主线秘密
- 揭露方式合理：通过行动后果暴露，而不是靠系统公告式说明
- 揭露后果已部分建立：`c002` 判断变化、局势目标变化、风险变化

结论：

- 信息释放成立
- 可视为 `valid_with_dependency`
- 当前不生成 `information_leak`

### 5.6 角色一致性审查

调用依据：

- `03_rules/03_character_consistency_rules.md`

当前判断如下：

- `c001` 的行为成立：她优先级前两位是活下去、隐藏秘密；在高压下独自先手夺令，符合其“独扛”的缺陷模式，也没有突然学会求助或完全信任任何人
- `c002` 的行为成立：他没有突然明确站队，只是从观察位置被逼到了半知情位置，符合“延迟决断”的模式
- `c003` 的行为成立：继续试探、施压、逼近异常，没有突然降智或情绪化失控
- 关系变化成立：`c001 ↔ c002` 没有一步跳到信任，只是进入更复杂、更危险的半合作边缘
- 成长弧没有突变：`c001` 没有突然成熟，`c002` 没有突然彻底站队

结论：

- `character_consistency_status = valid_with_cost`
- 但如果后续马上写成 `c001` 完全信任 `c002`，就会触发 `motivation_gap` 或 `relationship_jump`

### 5.7 承诺与结构审查

调用依据：

- `03_rules/06_foreshadow_rules.md`

当前判断如下：

- 主线推进成立：`fg_001` 残魂 / 身世主线得到推进，但没有过早揭晓
- 关系承诺成立：`fg_002` 的“`c002` 是否保密”被正式激活
- 新承诺合理：新开的都是当前局势自然生出的承诺，如 `c002` 的立场、令牌异常流出背后的力量
- 回收节奏成立：当前主要是在推进，而不是在结算，没有突兀回收

结论：

- 当前承诺管理成立
- 无正式 `abrupt_payoff`
- 无正式 `promise_loss`
- 但如果后续连续多个单元都不推动 `c002` 的立场问题或残魂线，会逐渐逼近 `promise_loss`

## 6. 审查结果汇总

### 6.1 PlotUnit 审查字段回写建议

```json id="exrv06plotwriteback"
{
  "progression_strength": "strong",
  "legality_status": "valid_with_cost",
  "character_consistency_status": "valid_with_cost",
  "redundancy_risk": "low",
  "review_notes": [
    "本单元是强推进单元，成功把资源争夺转化为撤离、封口与关系压力。",
    "后续 1 到 2 个单元必须兑现旧伤、追踪风险与保密压力，否则会升级为 missing_cost / missing_consequence。",
    "当前关系推进成立，但不允许在没有桥接的情况下直接进入稳定信任。"
  ]
}
```

### 6.2 失败类型判定

当前正式失败类型：

- 无

当前预警型潜在失败：

- `missing_cost`
- `missing_consequence`
- 远期可能的 `promise_loss`

主失败类型与次失败类型当前都为空。

### 6.3 是否生成 ReviewIssue

结论是：当前不生成阻断型正式 `ReviewIssue`。

原因很简单：

- 没有已经成立的硬错误
- 没有已经发生的角色失真
- 没有已经发生的承诺崩坏

但可以保留一个提醒型审查备注，作为近程续写约束。

```json id="exrv06reminder"
{
  "issue_type": "missing_consequence",
  "severity": "low",
  "status": "acknowledged",
  "summary": "scene_014 已激活旧伤、追踪与保密压力，后续 1-2 个单元内必须兑现至少其一。",
  "notes": "当前不阻断，只作为近程续写约束。"
}
```

### 6.4 最终审查结论

结论等级：

**通过但有警告**

理由如下：

1. 当前推进是有效的强推进
2. 世界合法性成立，但代价仍需兑现
3. 角色一致性成立，但关系线不能跳
4. 承诺推进成立，但后续需要跟上

### 6.5 后续动作建议

建议动作：

- 允许继续推进

附加条件：

下一轮 `Continue Workflow` 必须优先处理以下至少一项：

1. 旧伤后果
2. 巡查 / 追踪压力
3. `c002` 是否保密的立场动作

不建议直接做的事：

- 立即完整揭露残魂真相
- 立即让 `c001` 完全信任 `c002`
- 立即让局势跳到全新无关支线

## 8. 为什么这个审查样例是好的样例

这份样例有三个明显优点。

### 8.1 它不是“挑刺型审查”

它不是为了硬找问题，而是如实区分：

- 什么已经成立
- 什么还没坏
- 什么正在逼近风险

真实审查不是越严越好，而是越准确越好。

### 8.2 它能体现“通过但有警告”

很多系统只会输出“通过 / 不通过”，但真实工作流里大量结果其实是：

- 现在还能放行
- 但短期内必须补偿

这个层级非常关键。

### 8.3 它已经形成闭环

这次审查已经能完成：

1. 读取对象
2. 分项判断
3. 汇总结论
4. 判断是否建单
5. 给出后续动作

这说明 `Review Workflow` 在样例层已经能跑通。

## 9. 最小验证清单

用这份样例时，至少验证下面这些问题：

1. 系统能不能按顺序完成一次正式审查？
2. 系统能不能明确说出哪些地方成立、哪些地方只是警告？
3. 系统能不能不滥发 issue？
4. 系统能不能把审查结果回写到 `PlotUnit`？
5. 系统能不能根据这次审查决定下一步该怎么走？

如果这 5 个问题都能回答，说明这份样例是可用的。

---

## 10. 样例结论

这份 `sample_review` 的核心价值，不在于它“发现了多少问题”，而在于它已经展示了：

- 如何从对象层出发做审查
- 如何区分成立、警告、失败
- 如何控制 issue 的粒度
- 如何把审查结果真正写回系统
- 如何让审查成为后续续写和改写的入口

---

## 11. 一句话总结

这份 `sample_review` 的作用，是给 mock project 提供一个不是泛泛评价，而是能真正读取对象、分项判断、形成结论、决定放行与否的正式审查样例。
