# 示例：CharacterModel arc_stage 边界压测

## 文档目的

本文档提供一个 `arc_stage` 边界样例，用于验证：

- 哪些变化已经构成成长阶段推进
- 哪些变化仍只是“看起来像成长”的单次表现
- `arc_stage` 不应被写成每次情绪好转、一次忍耐成功或一次正确选择的流水账

它验证的不是：

- 角色每做一次重要决定都要不要推进成长阶段

它验证的是：

- `arc_stage` 作为可成长字段，何时应更新，何时仍应留在当前工作面或修复层

本文档主要对应：

- `02_data_models/03_charactermodel_schema.md`
- `03_rules/03_character_consistency_rules.md`
- `02_data_models/09_field_rules.md`
- `04_workflows/01_rebuild_workflow.md`

---

## 1. 测试定位

`arc_stage` 和 `self_image` 很像，但更容易出现另一种误判：

- 角色这次终于做对了
- 角色这次终于求助了
- 角色这次没有退回旧模式

于是系统很容易立刻把成长阶段往前推一格。

问题在于，很多这类变化仍可能只是：

- 一次高压下的例外动作
- 一次被逼出来的暂时偏移
- 一次局部补桥后的短时表现

如果这层不压，系统最容易出现两种错误：

1. 过早推进  
   一场戏后就把 `arc_stage` 从“未觉醒”推进到“阶段性突破”
2. 过迟推进  
   多轮选择、代价与回退都已经发生，`arc_stage` 却仍停在旧阶段

---

## 2. 测试前提

沿用当前 mock case，并假设：

```json
{
  "character_id": "c001",
  "arc_type": "认知修正",
  "arc_start": "把独自承担视为唯一生存方式",
  "arc_target": "承认信任与共同承担并非软弱",
  "arc_stage": "未觉醒"
}
```

后续 repeated `Continue / Review / Rebuild` 中，系统遇到以下三类变化。

### 2.1 候选变化 A

在一次危机里，`c001` 被迫把部分令牌信息告诉 `c002`，以换取即时脱身。

### 2.2 候选变化 B

在后续多轮里，`c001` 不只一次，而是反复：

- 主动共享关键信息
- 接受共同决策
- 在失败后仍未退回“只能独活”的旧模式
- 因此付出新的关系与信任成本

### 2.3 候选变化 C

一次短暂合作后，文本表面上把 `c001` 写得像已经“学会信任别人”，但下一轮又立刻退回旧行为基底。

---

## 3. Boundary 场景 A：一次被逼出来的偏移，不应直接推进 `arc_stage`

### 3.1 情况说明

候选变化 A 很容易被误判成成长节点，因为角色确实做了以前不会做的事：

- 向他人共享信息

但这一动作发生在：

- 被迫情境下
- 为了即时逃生
- 尚未证明会改变后续行为基底

### 3.2 正确裁定

这类变化默认不应直接更新 `arc_stage`。

更合适的去向是：

- `PlotUnit.state_change_summary`
- `NarrativeState` 当前合作方式
- 必要时 `Review` 的成长观察项

### 3.3 为什么这时不该推进

因为它只能证明：

- 角色出现了一次偏移

还不能证明：

- 角色已经进入新的稳定成长阶段

这更接近：

- stage evidence

而不是：

- stage transition

---

## 4. Boundary 场景 B：跨多轮、带代价且可回看成立的变化，可以推进 `arc_stage`

### 4.1 情况说明

候选变化 B 不只是一次例外动作，而是形成了 repeated pattern：

- 多轮选择一致
- 有明确触发与代价
- 失败后没有立刻全面退回旧模式
- 与 `arc_target` 的方向一致

### 4.2 正确裁定

这时可以推进 `arc_stage`，例如：

- 从 `未觉醒`
- 推进到 `开始怀疑只能独活的旧信念`

或者：

- 从 `开始怀疑`
- 推进到 `第一次稳定偏移`

### 4.3 为什么这次应推进

因为这已经不是：

- 单次成功表现

而是：

- 可回看的阶段性变化

它满足了成长弧审查里最关键的条件：

- 有触发
- 有代价
- 有重复性
- 有方向性

---

## 5. Boundary 场景 C：表面成长但快速回退，不应假装阶段已推进

### 5.1 情况说明

候选变化 C 表面上看像成长完成：

- 角色说出成熟的话
- 关系暂时变好
- scene 收尾像是完成了一个节点

但下一轮马上出现：

- 重新封闭
- 拒绝共享
- 把所有风险再次独吞

### 5.2 正确裁定

这类情况默认不应直接推进 `arc_stage`。

如果已经误推了，也应在 `Rewrite` / `Review` 后回退。

### 5.3 为什么这一步重要

因为成长阶段不应由：

- 一次看起来完整的 scene 结尾

来定义，而应由：

- 跨轮保持的行为基底变化

来定义。

否则 `arc_stage` 会变成：

- 文本气氛标签

而不是：

- 角色成长结构接口

---

## 6. Boundary 场景 D：不能把成长依据直接写成 `arc_stage`

### 6.1 错误写法

即使场景 B 满足推进条件，也不应把 `arc_stage` 写成：

- “最近三轮都开始愿意共享风险并付出信任代价”

### 6.2 正确裁定

应保留：

- 结构化阶段标签，如 `开始怀疑`、`第一次稳定偏移`

应迁出：

- “最近三轮……”
- 具体哪三轮
- 每次代价明细

这些更适合留在：

- handoff supporting context
- `ReviewIssue` / review note
- 相关 `PlotUnit`

### 6.3 为什么不能混写

否则 `arc_stage` 会从：

- 可复用的阶段接口

滑成：

- 带因果注释的成长流水账

---

## 7. 最小 keep-vs-drop 裁定表

| 候选内容 | 正确去向 | keep / drop | 核心理由 |
|---|---|---|---|
| 一次被逼出来的信息共享 | `PlotUnit` / `NarrativeState` / review observation | drop from `arc_stage` | 单次偏移，不足以证明阶段推进 |
| 多轮重复、带代价且未回退的共同承担 | `CharacterModel.arc_stage` | keep | 已形成可回看的阶段性变化 |
| 表面成长后下一轮立刻回退 | 当前工作面或修复层 | drop / rollback | 看起来像成长，不等于阶段成立 |
| “最近三轮都……”这类推进依据 | handoff / review supporting context | drop from `arc_stage` | 这是阶段依据，不是阶段标签 |

---

## 8. 这份样例验证了什么

这份样例主要验证五件事：

1. `arc_stage` 不等于每次成长表现都更新
2. 成长阶段推进必须有触发、代价、重复性和方向性
3. 单次偏移与阶段推进必须分开
4. 表面成长但快速回退不应被误判为阶段成立
5. `arc_stage` 应保持为结构化阶段接口，而不是成长过程流水账

---

## 9. 一句话总结

`arc_stage` 只该在跨多轮、带代价且可回看的行为基底变化出现时推进，而不该因为一次高压下的正确选择或一个看起来完整的成长 scene 就立刻前移。
