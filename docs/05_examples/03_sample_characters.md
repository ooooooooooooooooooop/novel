# 示例：CharacterModel

## 文档目的
本文件提供一组 `CharacterModel` 样例，用于验证：

- `CharacterModel Schema` 是否足以表达“角色作为决策体”的结构
- 角色模型是否真的能支撑状态转移、续写选择、关系推进与审查判断
- 角色的目标、恐惧、缺陷、知情范围、关系与成长阶段，是否能在后续 `PlotUnit` 中被调用，而不是只停留在人设描述层

本文件对应：

- `01_concepts/03_charactermodel.md`
- `02_data_models/03_charactermodel_schema.md`
- `03_rules/03_character_consistency_rules.md`

---

## 1. 示例定位

这组样例延续前面的统一 mock project：

- 作品：`残雪归宗`
- 世界：`北衡界`
- 当前阶段：试炼前夜
- 关键事件：雨夜夺令
- 核心主线：回宗、残魂秘密、旧案、`c002` 是否会保密

这份文件不追求“人物很多”，而追求：

- 主要角色够清晰
- 关系张力够清晰
- 行动逻辑够可调用
- 后续能直接进入 `NarrativeState`、`PlotUnit` 与 `Review Workflow`

---

## 2. 使用场景

这组样例主要用于验证以下问题：

### 2.1 角色模型能不能解释行为

系统能不能回答：

- 这个角色为什么这么做
- 他最可能优先保什么
- 他最怕什么
- 他会在哪些压力下偏移

### 2.2 关系字段能不能支撑剧情

系统能不能回答：

- 谁更信任谁
- 谁对谁有亏欠
- 谁会保护谁
- 谁在利用谁

### 2.3 成长字段是不是有用

系统能不能判断：

- 角色是不是突然成熟
- 角色是不是突然失真
- 某次关系变化是不是跳了桥

---

## 3. 示例角色清单

当前样例提供 3 个核心角色：

1. `c001 / 沈青`
2. `c002 / 顾临渊`
3. `c003 / 裴照`

它们的定位分别是：

- `c001`：主角 / 当前行动焦点 / 成长弧主体
- `c002`：关键关系节点 / 观察者 / 潜在保密者
- `c003`：压力制造者 / 误导制造者 / 风险放大器

## 5. 样例解读

### 5.1 为什么只选这三个人

当前样例不追求人物全，而追求“核心冲突最小闭环”。

这三个人已经足够形成：

- 主角生存与成长线：`c001`
- 关键关系与立场摇摆线：`c002`
- 压力、误导、竞争线：`c003`

这意味着在不扩充人物表的情况下，系统已经能跑出：

- 目标推进
- 风险升级
- 信息暴露
- 关系转化
- 承诺推进

这对 mock case 已经足够。

### 5.2 `c001 / 沈青` 为什么成立

她不是“有设定的主角”，而是一个明确的决策体。

她的行动会持续被四个优先目标牵动：

1. 活下去
2. 隐藏秘密
3. 回宗
4. 查真相

这组排序决定了她为什么会先处理生存，再处理关系、公正和追查。她的恐惧不是泛泛的“冷”，而是害怕再次被抛弃，害怕一旦依赖别人就会失控，因此她的缺陷表现为关键时刻独扛、不求助。

她的成长弧也不是“从弱变强”，而是从把所有关系都当危险源，逐步走向有限信任与承担后果。这使她能稳定支撑当前项目关心的成长、代价与身份认同主题。

### 5.3 `c002 / 顾临渊` 为什么成立

他不是单纯“好人导师”，而是知情边缘人、风险评估者、关系摇摆者和潜在保密者的复合体。

他的核心矛盾在于：想要真相，但又不想太早失去自己的行动空间。因此他的典型行为逻辑是先观察、先验证、先留后手，再决定是否真正站队。

他的缺陷不是冲动，而是拖延决断、过度依赖观察者位置。这会自然制造很多剧情空间，例如替沈青隐瞒却不彻底表态，保护她却保持警惕，关系推进时也不会一步到位。

他的成长弧是学会在证据不足时，也承担立场选择的后果。这正好与 `c001` 的信任弧互相咬合。

### 5.4 `c003 / 裴照` 为什么成立

他不是纯反派模板，而是试探者、压力制造者、假线索投放者和风险放大器。

他真正追求的不是“打败主角”，而是借旧案与试炼这条线完成自己上位。因此他不是挡路怪，而是会主动操控局势的人。

他的缺陷是过度相信自己能控场。这意味着他会持续加压、制造误导，并把所有人都算进自己的棋局里，但最终很可能被关系和情感后果反噬。

他承担的是堕落线，而不是成长线。这一点很重要，因为系统不应该让所有重要角色都只有“成长”。

### 5.5 三人关系结构为什么够强

当前三人关系不是平均分布，而是形成一个清晰的三角结构：

- `c001 ↔ c002`：信任摇摆、保护与隐瞒并存、亏欠感、未明说的依赖
- `c001 ↔ c003`：高压试探、利用与反利用、敌意升级、风险放大
- `c002 ↔ c003`：彼此都知道对方没说实话、表面平衡、随时可能互相踩线

这三条线放在一起，已经足以支撑关系推进单元、信息暴露单元、风险升级单元与试探误导单元。

### 5.6 为什么这些字段能支撑续写

这组 `CharacterModel` 样例不是为了“角色档案好看”，而是为了续写时能直接调用。

系统需要能据此回答：

- 对 `c001`：她现在最优先保什么，她更可能独扛还是求助，什么压力会逼她突破边界
- 对 `c002`：他会不会保密，会不会继续观察，什么条件下才会真正站队
- 对 `c003`：他下一步是毁掉主角还是继续利用，她最怕暴露什么，他会不会改用公开施压

如果这些问题能稳定回答，就说明 `CharacterModel` 真正能用于工作流。

### 5.7 为什么这些字段能支撑审查

在审查里，这组三人模型也很有用。

例如：

- 如果 `c001` 突然完全信任 `c002`，可能触发 `motivation_gap` 或 `relationship_jump`
- 如果 `c002` 突然公开站队沈青，可能触发 `character_distortion`，除非前文已经给出足够压力与证据
- 如果 `c003` 突然变成纯情绪化莽撞对手，也可能触发 `character_distortion`

这说明角色模型已经可以作为审查依据，而不是只做展示用。

### 5.8 为什么知情字段很关键

这份样例里，`knowledge_state`、`misinformation`、`unknown_critical_info` 都写得比较细。这是故意的，因为后续很多问题都发生在知情边界上：

- 角色知道了不该知道的
- 角色忽略了明明知道的
- 角色误解已经该解除却没解除

以 `c002` 为例，他知道沈青身上有异常痕迹，但不知道残魂的真实本质。因此他的合理行为应该是半信半疑、谨慎保护，而不是直接说出完整真相。知情字段写细，审查时才抓得住越界。

### 5.9 为什么运行态字段不是摆设

这组样例里有一些字段是明显会变化的：

- `short_term_goal`
- `available_resources`
- `knowledge_state`
- `relations`
- `arc_stage`

这意味着角色模型不会僵死，而会随着续写工作流不断更新。

例如 `c001.short_term_goal` 现在是“带着令牌活着撤离”，下一场就可能变成“决定是否向 `c002` 局部坦白”或“甩开追踪”；`c002.arc_stage` 现在是“开始怀疑自己无法继续中立”，一旦他做出保密选择，就可能继续推进。

这说明角色模型已经具备“可运行性”。

## 6. 最小验证清单

用这组样例时，至少验证下面这些问题：

1. 系统能不能根据这些模型判断三个人下一步最可能怎么行动？
2. 系统能不能根据这些模型判断什么行为会构成人设失真？
3. 系统能不能看出哪条关系线最重要？
4. 系统能不能根据这些模型生成一个合理的下一 `PlotUnit`？
5. 系统能不能根据这些模型识别什么叫“关系跳变”“动机不足”“知情越界”？

如果这 5 个问题都能回答，说明这组样例是可用的。

## 7. 样例结论

这组 `sample_characters` 的核心价值，不在于角色多，而在于它已经能提供：

- 可执行目标结构
- 可审查恐惧与缺陷
- 可推进关系张力
- 可管理知情边界
- 可追踪成长阶段

这就足以支撑后面的：

- `sample_plotunit`
- `sample_factledger`
- `sample_review`
