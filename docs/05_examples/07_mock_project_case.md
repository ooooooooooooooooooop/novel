# 示例：Mock Project Case

## 文档目的
本文件提供一个贯穿式 mock case，用于验证：

- 这套自动小说系统是否能从顶层规格一路落到运行态对象
- 各对象之间是否能真正形成闭环，而不是各写各的
- 工作流是否能在一个最小项目上跑通：重建、续写、审查、改写
- 规则层是否能在真实样例中发挥作用
- 前面几份 example 文档是否能被整合成一套最小可运行项目

本文件对应：

- `05_examples/01_sample_workspec.md`
- `05_examples/02_sample_worldmodel.md`
- `05_examples/03_sample_characters.md`
- `05_examples/04_sample_plotunit.md`
- `05_examples/05_sample_factledger.md`
- `05_examples/06_sample_review.md`
- `05_examples/08_reviewreminder_chain.md`

以及：

- `04_workflows/01_rebuild_workflow.md`
- `04_workflows/02_review_workflow.md`
- `04_workflows/03_continue_workflow.md`
- `04_workflows/04_rewrite_workflow.md`

## 1. 示例定位

这不是单一对象样例，而是一个最小全链路项目样例。

它的目标不是写完一部长篇小说，而是验证：

1. 一个项目如何初始化
2. 一个当前状态如何被建立
3. 一次推进如何发生
4. 一次审查如何落地
5. 一次问题如何进入改写闭环

一句话定义：

`mock_project_case` 是这套系统的第一块全链路试跑地。

---

## 2. 案例总览

### 2.1 项目名

`残雪归宗`

### 2.2 当前阶段

第一卷中前段，试炼前夜。

### 2.3 核心前提

被逐出宗门的少女沈青，必须在三年内重返仙门，否则体内残魂将逐步吞噬她。

### 2.4 当前局势

在黑水泽外围的废弃驿亭中，围绕一枚试炼令牌，三人局势被同时推高：

- `c001 / 沈青`：必须拿到令牌并活着撤离
- `c002 / 顾临渊`：已经开始确认她身上的异常，但尚未完全站队
- `c003 / 裴照`：持续试探、施压，试图把她逼到暴露边缘

### 2.5 当前核心承诺

- 主角的身世与残魂秘密
- 被逐旧案背后的人
- `c002` 最终会保密、利用还是揭发
- 令牌异常流出背后的力量

---

## 3. 这个 mock case 要验证什么

这个案例主要验证五个问题：

### 3.1 顶层规格能不能约束下层

`WorkSpec` 是否真的影响世界模型、角色模型、推进方式与审查标准。

### 3.2 世界与角色能不能共同制造推进

系统能不能不是“凭空推进”，而是由角色目标、世界约束和自然冲突共同推动。

### 3.3 PlotUnit 能不能真正改变状态

不是发生了一场戏，而是目标、风险、信息与关系都真的发生变化。

### 3.4 审查能不能不空转

系统能不能具体指出什么成立、什么只是警告、什么需要建 issue。

### 3.5 改写是否真的有入口

如果后续某一步写坏了，系统能不能知道该修哪一层、该怎么修、修完要复检什么。

## 4. 项目对象总览

### 4.1 WorkSpec

已有样例：

- `05_examples/01_sample_workspec.md`

当前项目关键信息：

- 类型：仙侠 / 宗门成长 / 身世之谜 / 女性成长
- 主题：成长、代价、命运、身份认同
- 调性：克制
- 叙述：限制第三人称
- 节奏：前快中稳后爆

它在本案例中的作用，是约束作品不要突然漂向纯爽文推进、全知乱飞或脱离成长线的空转感情线。

### 4.2 WorldModel

已有样例：

- `05_examples/02_sample_worldmodel.md`

当前关键规则：

- 灵根受损不可自然恢复
- 禁术使用会留下可追踪痕迹
- 王城内禁止公开斗法
- 试炼资格受宗门控制
- 宗门、王城、世家三方制衡

它在本案例中的作用，是确保沈青不能无代价爆发、令牌不能像普通道具一样乱发、异常灵压外泄必须产生后果、旧案与监察线能够持续追上来。

### 4.3 CharacterModel

已有样例：

- `05_examples/03_sample_characters.md`

当前核心角色：

- `c001 / 沈青`
- `c002 / 顾临渊`
- `c003 / 裴照`

它们共同决定谁推动局势、谁维持风险平衡、谁制造误导，以及关系线为什么不能一步跳完。

### 4.4 PlotUnit

已有样例：

- `05_examples/04_sample_plotunit.md`

当前关键单元：

- `pu_scene_014 / 雨夜夺令`

它承担当前最小强推进，把局势从“争夺资源”转向“撤离与封口”，同时激活 `c002` 保密线并推进残魂秘密线。

### 4.5 FactLedger

已有样例：

- `05_examples/05_sample_factledger.md`

当前关键事实：

- `f_014`：令牌归 `c001` 所有
- `f_015`：`c002` 已确认异常痕迹
- `f_016`：旧伤加重
- `f_017`：巡查风险上升
- `f_018`：`c001` 与 `c002` 进入高风险半合作边缘
- `f_019`：主目标转为撤离与封口

它约束后续不能装作什么都没发生，知情状态不能乱写，代价和追踪也必须兑现。

### 4.6 Review

已有样例：

- `05_examples/06_sample_review.md`

当前结论：

- `pu_scene_014`：通过但有警告

当前警告重点是，后续 1 到 2 个单元内必须体现旧伤、追踪风险与保密压力；这一点已由 `ReviewReminder` 进入正式轻量追踪层。

## 5. 从输入到当前状态的最小链路

下面用一条最简链路把前面所有对象串起来：

```text
WorkSpec
  ↓
决定作品方向与约束
  ↓
WorldModel
  ↓
决定世界规则、资源、制度与后果
  ↓
CharacterModel
  ↓
决定谁在当前局里会怎么选
  ↓
NarrativeState（scene_014 前）
  ↓
定义“现在的局”
  ↓
PlotUnit（雨夜夺令）
  ↓
造成资源、关系、信息、风险变化
  ↓
FactLedger / ForeshadowGraph 更新
  ↓
NarrativeState（scene_015）
  ↓
Review Workflow 审查
  ↓
决定可继续推进
```

这条链路能跑通，说明系统至少已经具备最小运行能力。

## 6. 当前最小项目快照

### 6.1 项目级快照

作品目标：

写一个以“回宗、生存、身份与信任”为核心的女性成长仙侠故事。

当前卷目标：

把主角从“被逐边缘人”推进到“重新拥有进入核心棋局资格的人”。

当前段目标：

让她先拿到令牌，再被迫面对：

- 谁能信
- 谁在追
- 自己到底暴露到了哪一步

### 6.2 世界级快照

当前最重要的世界压力：

1. 禁术痕迹可追踪
2. 试炼资格受控
3. 旧案会触发监察司
4. 身份无保护时，风险会迅速升级

当前最重要的环境压力：

- 黑水泽外围暴雨
- 巡查路线变化
- 驿亭不是安全地带

### 6.3 角色级快照

`c001`

- 最想：活着带令撤离
- 最怕：秘密被确认后再次被当成祸物抛弃
- 当前缺陷：宁可独扛也不愿求助

`c002`

- 最想：在不彻底失去行动空间的前提下掌握真相
- 最怕：站错边，把所有人一起送进去
- 当前缺陷：拖延立场决断

`c003`

- 最想：利用当前局势为自己上位
- 最怕：再被更高层当成弃子
- 当前缺陷：高估自己的控场能力

### 6.4 局势级快照

当前 `NarrativeState` 压缩版如下：

- 时间：试炼开始前夜
- 地点：黑水泽外围废弃驿亭
- 主目标：带令撤离并控制秘密扩散
- 主冲突：巡查逼近、`c002` 半知情、`c003` 仍具威胁
- 当前情绪温度：压抑、戒备、敌对
- 当前开放问题：`c002` 会不会保密、`c003` 背后是谁、追踪风险多久会落地

## 7. 本案例如何跑通四条工作流

### 7.1 Rebuild Workflow

适用想象场景：

假设手里只有章节 1 到 14、一些人物设定和一些世界观说明。

系统应能从这些文本中重建：

- `WorkSpec`
- `WorldModel`
- 三个主要 `CharacterModel`
- 当前阶段关键 `FactLedger`
- 两条主承诺线程
- 当前 `NarrativeState`

重建完成的标志是系统已经能回答：

- 现在的局是什么
- 下一步最该推进什么
- 哪些秘密已半暴露
- 哪些后果已经该追上来

### 7.2 Continue Workflow

当前输入：

- `s_015_scene`
- 相关角色模型
- 当前账本
- 当前承诺图

根据当前样例，下一步最合理的推进方向大致应是：

1. 兑现旧伤、追踪或保密后果
2. 推进 `c002` 的立场动作
3. 建立新的撤离局势
4. 继续推进主线谜团，但不能直接揭晓

可生成的下一个 `PlotUnit` 例如：

- `夜雨撤离`
- `临时同盟`
- `巡查逼近下的假路线误导`

### 7.3 Review Workflow

当前对象：

- `pu_scene_014`

当前审查结果：

- 强推进
- 世界合法
- 角色一致
- 承诺推进成立
- 但必须在短期内兑现代价与后果

因此当前决策是：

- 可放行
- 带警告继续推进

### 7.4 Rewrite Workflow

假设未来出现问题，例如下一场直接写成：

- `c001` 完全信任 `c002`
- 两人毫无保留联手逃离
- 且没有任何旧伤或保密压力

系统应先通过审查抓到：

- `relationship_jump`
- `motivation_gap`
- `missing_consequence`

然后进入改写工作流：

1. 读取 `ReviewIssue`
2. 定位问题主要在 `PlotUnit` 与 `NarrativeState`
3. 选择修法，如 `relationship_bridge_scene` 或 `character_motivation_patch`
4. 改写为不是完全信任，而是被迫形成有限合作
5. 复检并关闭 issue

这说明 mock case 已经能支撑“坏了以后怎么修”。

## 8. 一轮最小运行示例

下面给出一个最小运行版本，展示这套系统在这个案例里怎么动。

### Step 1：已有对象层

系统已经有：

- `WorkSpec`
- `WorldModel`
- `CharacterModel`
- `FactLedger`
- `ForeshadowGraph`
- 当前 `NarrativeState`

### Step 2：生成 PlotUnit

根据当前主驱动力：

- 巡查逼近
- 旧伤
- 保密压力

系统生成：

- `pu_scene_015 / 夜雨撤离`

### Step 3：更新对象

更新：

- 新 `NarrativeState`
- 新 `FactLedger` 条目
- `fg_002` 的推进节点
- 必要时 `c001` 与 `c002` 的 `relations`

### Step 4：审查

运行 `Review Workflow`。

### Step 5：分流

- 若通过：继续下一轮
- 若失败：进入 `Rewrite Workflow`

这就是最小一轮系统闭环。

## 9. 这个 mock case 目前已经验证了什么

到这里，这个案例已经能验证：

### 9.1 对象层不是空架子

前面定义的对象都能在同一个案例里找到落点。

### 9.2 Schema 不是摆设

字段真的能填，而且填完后能驱动后续动作。

### 9.3 规则层能用

不是只会写原则，而是能对这个案例做判断。

### 9.4 工作流能串起来

至少重建、续写、审查、改写已经有可运行路径。

## 10. 这个 mock case 还没有验证什么

虽然它已经很有用，但仍有几类东西还没有真正验证到。

### 10.1 多章节连续运行

现在主要验证的是一个中段单元及其前后，还没验证连续 5 到 10 个 `PlotUnit` 跑下来会不会漂。

### 10.2 多承诺线程并发

现在主要是两条重点承诺线程，还没验证 5 到 8 条线程同时活跃时系统会不会混乱。

### 10.3 大型改写

现在验证的是局部改写，还没验证 arc 级重规划是否真的可执行。

### 10.4 样例之外世界的迁移能力

现在所有样例都在同一个仙侠案例里，还没验证换成现代悬疑、科幻、都市是否仍成立。

这正是后续 `06_experiments/` 该做的事。

## 11. 用这个 mock case 检查整套系统的最小清单

当你想快速判断“这套系统现在是不是已经有雏形”，就看这八个问题：

1. 能不能从 `WorkSpec` 推出正确世界与角色方向？
2. `WorldModel` 能不能真的限制角色行为？
3. `CharacterModel` 能不能解释关键决策？
4. `PlotUnit` 能不能造成明确状态变化？
5. `FactLedger` 能不能稳定记录关键硬事实？
6. `ForeshadowGraph` 能不能抓住主承诺线程？
7. `Review Workflow` 能不能判断成立、警告、失败？
8. `Rewrite Workflow` 能不能对失败做最小修复？

如果这八个问题基本都能回答，说明这套系统已经不是空想了。

## 12. 当前建议的后续动作

这个 mock case 写完后，最自然的下一步不再是补更多概念，而是进入实验和决策记录层。

第一优先建议：

- `06_experiments/00_experiment_log.md`

因为你现在已经有了一个可试跑样例，最该做的是记录第一次试跑、记录哪里顺、哪里卡，以及哪些字段其实不够用。

第二优先建议：

- `07_decisions/00_decision_log.md`

因为从现在开始，很多修改不再是“想法”，而会变成：

- 为什么保留这个字段
- 为什么删掉那个字段
- 为什么工作流改顺序

## 13. 样例结论

这份 `mock_project_case` 的核心价值，不在于它把故事写完了，而在于它已经把：

- 作品规格
- 世界模型
- 角色模型
- 当前状态
- 推进单元
- 事实账本
- 承诺图
- 审查结果
- 改写入口

全部压进了同一个最小项目里。

这意味着你现在已经拥有了一块真正能拿来试跑、复盘、修订系统结构的第一块地。

## 14. 一句话总结

这份 `mock_project_case` 的作用，是把前面所有零散样例收束成一个可以被重建、可以被续写、可以被审查、也可以被改写的最小全链路项目样例。
