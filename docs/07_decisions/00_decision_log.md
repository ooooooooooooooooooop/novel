# 决策日志

## 文档目的
本文件作为自动小说系统的统一决策入口，用于记录每一次重要设计选择、规则取舍、字段增删、工作流调整与方向收束。

它回答的不是“系统现在长什么样”，而是：

- 为什么会长成这样
- 为什么选 A，不选 B
- 哪些方案被保留
- 哪些方案被放弃
- 哪些决定是临时性的
- 哪些决定已经变成当前地基

它不是实验记录，而是把已经做出的关键设计判断固化成可追踪、可引用、可防遗忘的项目决策。

---

## 1. 为什么必须有决策日志

到了当前阶段，这个项目已经不再只是零散想法，而是已经形成了：

- 项目层
- 概念层
- 数据模型层
- 规则层
- 工作流层
- 示例层
- 实验入口

这意味着后面会越来越频繁地遇到这些情况：

- 某个字段到底为什么存在
- 某条规则为什么写得这么严
- 为什么对象边界要这样切
- 为什么先做规则层，再做 workflow
- 为什么 `FactLedger` 不等于剧情摘要
- 为什么 `ForeshadowGraph` 和 `FactLedger` 必须分开
- 为什么当前选择 mock case 统一世界，而不是多题材并行

如果没有决策日志，后面很容易反复出现“忘了为什么这么定”“同一个问题重复争论”“把暂时方案误当最终方案”的情况。

所以决策日志的目标是把“已经想清楚的东西”固定下来，避免系统在后续扩展时反复失忆。

---

## 2. 与实验日志的区别

这两个文件容易混，但职责不同。

### 2.1 实验日志回答什么

- 试了什么
- 怎么试的
- 结果怎么样
- 发现了什么问题

### 2.2 决策日志回答什么

- 基于已有判断，最后决定怎么做
- 为什么这样做
- 这个决定影响哪些文件
- 这个决定是暂定还是正式采用

一句话区别：

- `06_experiments/` 更像证据流
- `07_decisions/` 更像裁定流

---

## 3. 适用范围

本日志适用于记录以下类型的决策：

### 3.1 对象边界决策

例如：

- `NarrativeState` 和 `FactLedger` 如何分工
- `ForeshadowGraph` 和 `FactLedger` 为什么不能混用
- `CharacterModel` 的哪些字段允许自动更新

### 3.2 字段设计决策

例如：

- 为什么某个字段保留
- 为什么某个字段删除
- 为什么不把某些派生指标直接写死进 schema

### 3.3 规则设计决策

例如：

- 为什么审查顺序先硬错误再角色再承诺
- 为什么 `missing_cost` 和 `missing_consequence` 分开
- 为什么信息释放必须单独成规则层

### 3.4 工作流设计决策

例如：

- 为什么先做 `Review Workflow`
- 为什么续写后必须立刻审查
- 为什么改写必须以 `ReviewIssue` 为入口

### 3.5 示例与实验策略决策

例如：

- 为什么先用一个统一 mock case
- 为什么先做正例，再补反例
- 为什么暂时不做多题材迁移验证

---

## 4. 核心原则

### 4.1 只记录已经做出的选择

不是所有想法都记进来。只有当一个问题已经有了阶段性答案，才值得写入决策日志。

### 4.2 决策必须有上下文

不能只写“决定保留这个字段”，至少要写清：

- 当时在解决什么问题
- 有哪些备选方案
- 为什么最后选了这个

### 4.3 决策必须有影响范围

每一条决策都最好标清它影响：

- 哪些目录
- 哪些文件
- 哪些对象
- 哪些后续步骤

### 4.4 区分暂定和固化

很多决策在当前阶段只是：

- 暂时采用
- 先这样走
- 等实验再确认

有些决策则已经变成：

- 当前地基
- 不轻易推翻的规则

这两类不能混着写。

---

## 5. 决策状态建议

建议每条决策至少标一个状态：

- `proposed`：提出但尚未正式采用
- `adopted`：已采用，作为当前默认方案
- `provisional`：暂定采用，后续仍需实验验证
- `deprecated`：曾采用，但现在不再推荐
- `rejected`：明确讨论过并决定不采用

---

## 6. 决策编号规则

建议统一使用：

```text
DEC-YYYYMMDD-序号
```

例如：

- `DEC-20260326-01`
- `DEC-20260326-02`

如果想区分类型，也可以扩展成：

```text
DEC-MODEL-20260326-01
DEC-RULE-20260326-02
DEC-WF-20260326-03
```

其中：

- `MODEL` = 数据模型决策
- `RULE` = 规则决策
- `WF` = 工作流决策
- `EX` = 示例 / 实验策略决策

---

## 7. 每条决策建议包含的内容

建议一条决策至少包含以下内容：

1. 决策编号
2. 决策标题
3. 决策状态
4. 背景问题
5. 备选方案
6. 最终选择
7. 选择理由
8. 影响范围
9. 风险或待验证项
10. 后续动作

---

## 8. 日志模板

下面给出推荐模板。

```md id="dec00template"
## DEC-YYYYMMDD-01

### 决策标题
一句话描述这次决定。

### 决策类型
- model / rule / workflow / example / project

### 决策状态
- proposed / adopted / provisional / deprecated / rejected

### 背景问题
这次决策在解决什么问题？

### 备选方案
- 方案 A：
- 方案 B：
- 方案 C：

### 最终选择
最终决定采用什么方案？

### 选择理由
为什么这么选？
可以写：
- 优点
- 代价
- 为什么不选其他方案

### 影响范围
影响哪些目录、文件、对象或流程？

### 风险 / 待验证项
这条决策还有哪些不确定性？

### 后续动作
- keep / patch / rewrite / re-evaluate

### 备注
补充说明。
```

## 9. 建议先记录的首批核心决策

基于你现在已经完成的内容，建议优先把以下决策写进去：

1. 系统先以“对象层 + 规则层”建地基，而不先实现代码系统
2. 核心对象固定为 8 个，不继续在当前阶段扩对象种类
3. `FactLedger` 与 `ForeshadowGraph` 严格分层
4. 规则层先完整写出，再补工作流层
5. 工作流先写 `Review Workflow`，再写 `Continue Workflow`
6. 示例层先统一使用一个 mock case，不做多题材并行
7. 当前阶段优先“地基成立”，不急于实现自动生成正文

## 10. 首条示例决策记录

```md id="dec00sample1"
## DEC-20260326-01

### 决策标题
当前项目优先构建“对象层 + 规则层 + 工作流层”的文档地基，而不直接进入代码实现

### 决策类型
project

### 决策状态
adopted

### 背景问题
项目最初的问题是：如果目标是制造一套能够自动写小说的系统，应该先从哪里开始。

可选路径包括：
- 直接写 prompt 和生成链路
- 直接做数据库 / schema / 程序实现
- 先做顶层架构文档
- 先做对象、规则和工作流定义

### 备选方案
- 方案 A：直接实现一个初版自动续写器
- 方案 B：先做部分 schema，再边做边补
- 方案 C：先把项目层、对象层、规则层、工作流层完整写出来，作为系统地基

### 最终选择
采用方案 C：先完成文档型系统地基，再考虑实现与自动化。

### 选择理由
1. 当前项目只有作者和 AI，没有管理层与团队协同负担，因此最重要的是先把思维结构稳定下来
2. 如果直接进入实现，极容易在对象边界、规则边界不清时反复推翻
3. 自动小说系统的真正难点不在“生成一句话”，而在“连续性、约束、审查与回写”
4. 先把地基文档写清，后续无论是给 Codex 看、给自己复用，还是做实现，都会更稳

### 影响范围
- `00_project/`
- `01_concepts/`
- `02_data_models/`
- `03_rules/`
- `04_workflows/`

### 风险 / 待验证项
- 文档地基可能过度抽象，后续仍需通过示例层和实验层验证可运行性
- 若样例层跑不通，需要回头修正当前抽象层

### 后续动作
keep

### 备注
这是当前阶段的总方向决策，后续很多子决策都以此为前提。
```

## 11. 第二条示例决策记录

```md id="dec00sample2"
## DEC-20260326-02

### 决策标题
将 FactLedger 与 ForeshadowGraph 严格拆分，不使用单一对象同时承载“事实”与“承诺”

### 决策类型
model

### 决策状态
adopted

### 背景问题
在设计中期，出现过一个自然诱惑：是否可以用一个统一对象同时管理：

- 已确认事实
- 悬念
- 伏笔
- 承诺
- 误导
- 回收状态

这样表面上看会更简单。

### 备选方案
- 方案 A：统一成一个“大记忆对象”
- 方案 B：事实与承诺分层，分别由 `FactLedger` 和 `ForeshadowGraph` 管理

### 最终选择
采用方案 B：严格拆分。

### 选择理由
1. “已成立事实”和“未来必须回应的承诺”不是同一类东西
2. 若混在一起，极容易把误导写成事实，或把承诺误当已成立状态
3. 审查时需要分别回答：
   - 有没有与事实冲突
   - 有没有承诺遗失 / 回收突兀
4. 续写时也需要分别处理：
   - 事实用于约束连续性
   - 承诺用于调度推进方向

### 影响范围
- `01_concepts/06_factledger.md`
- `01_concepts/07_foreshadowgraph.md`
- `02_data_models/06_factledger_schema.md`
- `02_data_models/07_foreshadowgraph_schema.md`
- `03_rules/05_information_release_rules.md`
- `03_rules/06_foreshadow_rules.md`

### 风险 / 待验证项
- 拆分后，两个对象之间的联动成本变高
- 后续需要通过样例和实验验证：这两个对象是否会重复记录、边界是否会模糊

### 后续动作
watch

### 备注
这是整个系统最重要的分层决策之一，后续不应轻易回退成“大一统记忆层”。
```

## 12. 第三条示例决策记录

```md id="dec00sample3"
## DEC-20260326-03

### 决策标题
工作流层优先实现 Review Workflow，而不是先写 Continue Workflow

### 决策类型
workflow

### 决策状态
adopted

### 背景问题
在进入 `04_workflows/` 时，存在一个选择：
- 先写“怎么继续推进”
- 还是先写“怎么判断推进是否成立”

### 备选方案
- 方案 A：先写 Continue Workflow
- 方案 B：先写 Review Workflow

### 最终选择
采用方案 B：先写 Review Workflow。

### 选择理由
1. 自动小说系统最容易失控的不是“写不出来”，而是“越写越漂”
2. 如果没有审查入口，续写工作流会缺少明确的放行标准
3. Review Workflow 可以作为后续 Continue / Rewrite 的共同枢纽
4. 审查逻辑已经在规则层比较成熟，先落成 workflow 更自然

### 影响范围
- `04_workflows/02_review_workflow.md`
- `04_workflows/03_continue_workflow.md`
- `04_workflows/04_rewrite_workflow.md`

### 风险 / 待验证项
- 先有审查不代表续写一定自然，后续仍需验证 Continue Workflow 是否真正受益于此顺序

### 后续动作
keep

### 备注
这是工作流层的顺序性决策，不是说 Continue Workflow 不重要，而是说它更适合建立在审查入口之后。
```

## 13. 当前阶段最重要的决策主题

如果只看当前阶段，最值得持续记录的决策主要集中在四类：

### 13.1 边界决策

例如：

- 哪个对象管什么
- 哪个对象不该管什么

### 13.2 粒度决策

例如：

- `FactLedger` 记到多细
- `PlotUnit` 最小粒度是什么

例如：

- 先写规则再写 workflow
- 续写后必须审查
- 改写必须由 issue 驱动

### 13.4 范围决策

例如：

- 当前先做一个统一 mock case
- 当前不扩多题材
- 当前不急于实现代码系统

## 15. 与其他目录的关系

### 15.1 与实验日志的关系

实验日志提供：

- 证据
- 现象
- 失败样式

决策日志提供：

- 结论
- 选择
- 固化

### 15.2 与规则层的关系

规则层写：

- 应该怎样

决策日志记录：

- 为什么这样写
- 为什么不用别的写法

### 15.3 与工作流层的关系

工作流层写：

- 实际怎么跑

决策日志记录：

- 为什么按这个顺序跑
- 为什么不按别的顺序跑

---

## 16. 当前阶段的使用建议

从现在开始，建议每当出现以下情况时，就补一条决策：

### 16.1 当一个争议已经被解决

例如两种方案选了一种。

### 16.2 当一个边界已经被固定

例如某对象职责已明确。

### 16.3 当一个顺序已经被确定

例如某 workflow 必须在另一条 workflow 之后。

### 16.4 当一个方案被明确放弃

例如：

- 不做多题材并行验证
- 不把事实与承诺混成一个对象
