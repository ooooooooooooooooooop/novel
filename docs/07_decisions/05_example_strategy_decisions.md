# 示例策略决策

## 文档目的
本文件用于收束当前阶段已经基本确定的示例层与验证策略相关决策。

它关注的不是某个样例具体写什么，而是更上层的问题：

- 为什么当前示例层这样组织
- 为什么先做这些样例，而不是那些样例
- 为什么当前先用一个统一 mock case
- 为什么先做正例，再补反例
- 示例层与实验层如何分工
- 什么时候应该引入第二题材、第二世界或更复杂样例

本文件的作用，是防止后续扩展时出现样例越来越多但没有验证重点、示例层和实验层职责混乱，以及过早追求多题材覆盖导致核心链路没跑通。

---

## 1. 文档定位

前面的决策文件已经固定了：

- 总方向
- 对象边界
- workflow 顺序
- schema 粒度

本文件固定的是：

样例应该怎样被组织，验证应该按什么策略推进。

它主要影响：

- `05_examples/`
- `06_experiments/`
- 后续可能新增的反例库
- 未来从 mock case 走向多题材样例的节奏

---

## 2. 当前需要固定的示例策略问题

当前最值得固定的示例策略问题主要有 8 个：

1. 为什么当前只使用一个统一 mock case
2. 为什么示例层先做正例而不是先做反例
3. 为什么示例层优先验证“闭环”，而不是“复杂度上限”
4. 示例层与实验层如何分工
5. 为什么当前样例以“对象联调”为核心，而不是以“文本质量展示”为核心
6. 什么时候应该引入第二个 mock case
7. 什么时候应该建立反例样例库
8. 当前样例优先服务 workflow 验证，而不是题材迁移验证

## 3. DEC-20260326-32

### 决策标题

当前先使用一个统一 mock case，不并行展开多题材样例。

### 决策类型

`example`

### 决策状态

`adopted`

### 背景问题

进入示例层后，有两种自然路径：

- 同时做仙侠、现代、悬疑、科幻等多个样例
- 先用一个统一案例把所有对象、规则和 workflow 串起来

### 备选方案

- 方案 A：多题材并行起步
- 方案 B：单一 mock case 先跑通全链路

### 最终选择

采用方案 B。

### 选择理由

当前统一案例是：

- 项目：`残雪归宗`
- 世界：`北衡界`
- 核心人物：`沈青 / 顾临渊 / 裴照`
- 关键场景：`雨夜夺令`
- 核心承诺：身世、残魂、保密、旧案

选择这一方案的原因是：

1. 同一案例最容易暴露对象之间是否能互相引用
2. 同一案例最容易跑通 rebuild、continue、review、rewrite
3. 当前最重要的是链路成立，不是覆盖率好看
4. 多题材过早展开会把问题来源混杂化

### 影响范围

- `05_examples/`
- `06_experiments/`
- 后续样例扩展节奏

### 风险 / 待验证项

- 统一案例可能让系统过度贴合仙侠语境
- 后续必须通过第二题材样例做迁移验证

### 后续动作

`watch`

## 4. DEC-20260326-33

### 决策标题

示例层先做正例闭环，再补反例压力测试。

### 决策类型

`example`

### 决策状态

`provisional`

### 背景问题

示例层的第一批样例，是应该一开始就同时做正例和反例，还是先把一条正向闭环搭起来，需要明确。

### 备选方案

- 方案 A：正例、反例同时铺开
- 方案 B：先做正例闭环，再通过实验层和后续样例补反例

### 最终选择

采用方案 B。

### 选择理由

当前正例范围已经优先搭建：

- `sample_workspec`
- `sample_worldmodel`
- `sample_characters`
- `sample_plotunit`
- `sample_factledger`
- `sample_review`
- `mock_project_case`

先这样做的原因是：

1. 当前首先需要证明“系统在正常案例下能成立”
2. 如果一开始就把大量精力投到反例，会打断对象与 workflow 的首次联调
3. 正例更适合帮助压实对象边界、基础字段和基本流程顺序

### 影响范围

- `05_examples/`
- `06_experiments/`

### 风险 / 待验证项

- 只有正例会让规则显得总是成立
- 后续必须尽快补关系跳变、动机不足、后果缺失、承诺遗失、信息越界等反例

### 后续动作

`patch`

## 5. DEC-20260326-34

### 决策标题

当前示例层优先验证闭环成立，不优先验证复杂度上限。

### 决策类型

`example`

### 决策状态

`adopted`

### 背景问题

示例层有两种建设目标：

- 闭环优先：确保一条最小项目链路能从头跑到尾
- 复杂度优先：一开始就压入大量人物、线程、阵营和时空切换

### 备选方案

- 方案 A：尽快做复杂样例，挑战上限
- 方案 B：先证明最小闭环成立

### 最终选择

采用方案 B。

### 选择理由

1. 如果最小闭环都跑不通，复杂样例只会掩盖问题
2. 当前阶段更需要回答“这套系统能不能最小运行”
3. 复杂度测试适合放到实验层作为第二阶段工作

当前闭环的最小定义至少包括：

1. `WorkSpec` 能约束下层
2. `WorldModel` 能约束行动
3. `CharacterModel` 能解释决策
4. `PlotUnit` 能造成状态变化
5. `FactLedger` 能约束连续性
6. `ForeshadowGraph` 能调度承诺
7. `Review` 能决定放行与否
8. `Rewrite` 能处理局部失败

### 影响范围

- 所有当前样例文件的设计尺度

### 风险 / 待验证项

- 闭环优先可能延后暴露复杂多线程、多阵营场景下的问题

### 后续动作

`keep`

## 6. DEC-20260326-35

### 决策标题

示例层负责可运行样例，实验层负责压力测试与反例验证。

### 决策类型

`example`

### 决策状态

`adopted`

### 背景问题

示例层和实验层很容易混成一层：都在写案例、都在写观察、都在验证规则。

如果不分工，后续会出现：

- 样例文件里塞满实验结论
- 实验日志里反复重写样例对象

### 备选方案

- 方案 A：样例和实验混用
- 方案 B：样例层提供可运行对象，实验层对这些对象做试跑与破坏性验证

### 最终选择

采用方案 B。

### 选择理由

示例层负责：

- 提供稳定、清晰、可引用的正向样例
- 帮助对象、规则、workflow 落地
- 形成项目级 mock case

实验层负责：

- 用样例试跑 workflow
- 制造反例
- 做边界压力测试
- 记录失败与结论

这样分工的好处是：

1. 样例层保持清晰与可复用
2. 实验层允许脏、快、带试错痕迹
3. 二者分开后，后续修改成本更低

### 影响范围

- `05_examples/`
- `06_experiments/`

### 风险 / 待验证项

- 若实验层长期不沉淀，可能导致有效反例长期停留在临时记录状态

### 后续动作

`keep`

## 7. DEC-20260326-36

### 决策标题

当前样例优先服务对象联调，不是为了展示“写得像小说”。

### 决策类型

`example`

### 决策状态

`adopted`

### 背景问题

样例层很容易不自觉滑向：

- 写得有画面感
- 写得像一段小说
- 写得很完整很好看

但这和当前项目阶段的真正目标并不一致。

### 备选方案

- 方案 A：把样例写成更像正文的展示
- 方案 B：把样例写成更像对象联调材料的运行样例

### 最终选择

采用方案 B。

### 选择理由

1. 当前目标是验证对象边界、字段是否够用、workflow 是否能跑
2. 如果优先追求文本展示感，容易弱化对象层严谨度
3. 正文表现力属于后续表达层问题，不是当前示例层核心指标

### 影响范围

- `05_examples/*`
- 后续实验口径

### 风险 / 待验证项

- 后续需要补一层“对象到文本”的桥接样例，但不是现在

### 后续动作

## 8. DEC-20260326-37

### 决策标题

只有当当前 mock case 能连续稳定跑两轮以上，才引入第二个 mock case。

### 决策类型

`example`

### 决策状态

`adopted`

### 背景问题

什么时候扩第二案例，是当前策略里一个关键分水岭。

### 备选方案

- 方案 A：现在就引入第二题材样例
- 方案 B：等当前 mock case 连续跑两轮以上后再扩

### 最终选择

采用方案 B。

### 选择理由

“连续稳定跑两轮以上”的最低定义至少包括：

1. 当前 mock case 已完成一轮 `Continue`
2. 一轮 `Review`
3. 至少一轮 `Rewrite` 或失败修复验证
4. 再次进入 `Continue`
5. 未出现对象层大面积回滚

这样做的原因是：

1. 第二案例的价值在于验证泛化
2. 若第一个案例自身还没稳定，第二案例只会扩大不确定性
3. 先跑深，再跑宽，更适合当前阶段

### 影响范围

- 示例扩张节奏
- 实验优先级

### 风险 / 待验证项

- 第一案例若长期停留在单轮闭环，会拖慢第二案例的进入时点

### 后续动作

`watch`

## 9. DEC-20260326-38

### 决策标题

反例样例库应在实验层先形成，再视情况上收为示例层正式资产。

### 决策类型

`example`

### 决策状态

`adopted`

### 背景问题

反例到底应不应该直接写进 `05_examples/`，需要明确。

### 备选方案

- 方案 A：直接在示例层建立大量 bad cases
- 方案 B：先在实验层用临时反例验证，稳定后再上收成正式反例样例

### 最终选择

采用方案 B。

### 选择理由

1. 反例通常带有强实验性质
2. 不是每个坏例都值得成为长期资产
3. 实验层更适合快速制造、快速放弃、快速迭代反例
4. 只有那些高复用、高解释价值的坏例，才值得转成正式反例样例

### 影响范围

- `06_experiments/`
- 未来可能新增的 `05_examples/bad_cases/`

### 风险 / 待验证项

- 上收标准目前还没有完全固定，需要后续实验支持

### 后续动作

## 11. 当前示例策略的总原则

基于以上决策，当前建议统一遵守以下原则：

1. 先深后宽
2. 先正后反
3. 先闭环后上限
4. 样例要可引用，实验要可试错
5. 样例优先服务 workflow
