# 基础决策

## 文档目的
本文件用于收束当前阶段已经基本确定、并将长期影响项目结构的基础决策。

它不是实验记录，也不是概念解释，而是把“目前这套系统最底层已经选定的设计方向”固定下来。

本文件主要回答：

- 当前项目先做什么，不做什么
- 这套系统当前的基本构成是什么
- 为什么要先建文档地基，而不是直接实现
- 为什么对象层、规则层、工作流层要按当前顺序展开
- 哪些设计现在属于阶段性地基，不建议轻易推翻

## 4. DEC-20260326-01

### 决策标题

先做文档地基，不直接实现代码系统。

### 决策类型

`project`

### 决策状态

`adopted`

### 背景问题

如果目标是制造一套能够自动写小说的系统，应该先从哪里开始。

可选路径包括：

- 直接写 prompt 和生成链路
- 直接做数据库和程序实现
- 先做 schema
- 先做概念、规则与工作流地基

### 备选方案

- 方案 A：直接写 prompt 和生成链路
- 方案 B：直接做数据库和程序实现
- 方案 C：先做 schema，再边做边补
- 方案 D：先做项目层、概念层、规则层、工作流层和示例层地基

### 最终选择

采用方案 D：先构建项目层、概念层、数据模型层、规则层、工作流层和示例层，暂不进入代码实现。

### 选择理由

1. 当前真正难的不是“生成一句”，而是“连续运行”
2. 如果边界没想清，直接实现只会更快把混乱固化
3. 文档地基更适合当前“作者 + AI”的单人推进模式
4. 后续无论交给 Codex、自己实现还是继续扩写，都需要这一层稳定抽象

### 影响范围

- 整个项目当前阶段的工作顺序
- 所有目录的展开方式

### 风险 / 待验证项

- 文档地基可能过度抽象，后续仍需通过示例层和实验层验证可运行性
- 若样例层跑不通，需要回头修正当前抽象层

### 后续动作

## 5. DEC-20260326-02

### 决策标题

核心对象固定为 8 个。

### 决策类型

`model`

### 决策状态

`adopted`

### 背景问题

如果要让系统稳定运行，需要先确定最小核心对象集。

- 对象太少，会丢结构
- 对象太多，会在早期失控

### 备选方案

- 方案 A：继续扩对象，例如单独增加主题图谱、关系图谱、风格图谱、章节容器对象
- 方案 B：先固定一组最小核心对象，优先验证闭环

### 最终选择

当前核心对象固定为：

1. `WorkSpec`
2. `WorldModel`
3. `CharacterModel`
4. `NarrativeState`
5. `PlotUnit`
6. `FactLedger`
7. `ForeshadowGraph`
8. `ReviewIssue`

### 选择理由

这 8 个对象已经足以覆盖：

- 顶层作品规格
- 世界规则
- 角色决策体
- 当前局势
- 推进单元
- 已成立事实
- 承诺与伏笔
- 审查问题对象

### 影响范围

- `01_concepts/`
- `02_data_models/`
- 所有 workflow 输入输出设计

### 风险 / 待验证项

- 连续多轮续写后，8 个对象是否仍够用，仍需实验确认
- 某些边界对象未来可能需要作为辅助层出现，但不应在当前阶段抢进核心层

### 后续动作

## 6. DEC-20260326-03

### 决策标题

`FactLedger` 与 `ForeshadowGraph` 严格分层。

### 决策类型

`model`

### 决策状态

`adopted`

### 背景问题

设计中期出现过一个诱惑：是否可以用一个统一对象同时管理已确认事实、悬念、伏笔、误导与回收状态。

### 备选方案

- 方案 A：用一个统一对象同时管理事实与承诺
- 方案 B：拆成两个对象，分别记录已成立事实与未来必须回应的线程

### 最终选择

采用方案 B：

- `FactLedger` 记录已成立、可约束后文的事实
- `ForeshadowGraph` 记录承诺、伏笔、误导与回收线程

### 选择理由

1. “已经算数的东西”与“未来必须回应的东西”本质不同
2. 若混在一起，容易把误导写成事实
3. 审查时需要分别回答：
   - 有没有事实冲突
   - 有没有承诺遗失或回收突兀
4. 续写时也需要分别调用：
   - 账本负责约束连续性
   - 承诺图负责调度推进方向

### 影响范围

- [06_factledger.md](d:\Desktop\novel\docs\01_concepts\06_factledger.md)
- [07_foreshadowgraph.md](d:\Desktop\novel\docs\01_concepts\07_foreshadowgraph.md)
- 对应 schema
- 信息释放规则
- 伏笔规则

### 风险 / 待验证项

- 拆分后，两个对象之间的联动成本变高
- 后续需继续验证：边界是否稳定、是否会重复记录

### 后续动作

`keep`

## 7. DEC-20260326-04

### 决策标题

规则层先完整展开，再写工作流层。

### 决策类型

`workflow`

### 决策状态

`adopted`

### 背景问题

在进入中层设计时，存在两种顺序：

- 先写 workflow，再补规则
- 先把规则写清，再把 workflow 串起来

### 备选方案

- 方案 A：先 workflow，后 rules
- 方案 B：先 rules，后 workflow

### 最终选择

采用方案 B：先完成规则层，再展开工作流层。

### 选择理由

1. 工作流只是操作顺序，前提是知道“什么算成立”
2. 如果没有规则层，workflow 很容易变成泛流程图
3. 审查规则、状态转移规则、失败类型先落地后，workflow 才有稳定判断标准

### 影响范围

- `03_rules/`
- `04_workflows/`

### 风险 / 待验证项

- 规则层先展开不代表 workflow 自动自然，仍需通过样例和实验验证真正可跑

### 后续动作

`keep`

## 8. DEC-20260326-05

### 决策标题

工作流先写 `Review Workflow`。

### 决策类型

`workflow`

### 决策状态

`adopted`

### 背景问题

进入 `04_workflows/` 时，存在两种自然顺序：

- 先写 `Continue Workflow`
- 先写 `Review Workflow`

### 备选方案

- 方案 A：先写 `Continue Workflow`
- 方案 B：先写 `Review Workflow`

### 最终选择

先写：

- [02_review_workflow.md](d:\Desktop\novel\docs\04_workflows\02_review_workflow.md)

再写：

- [03_continue_workflow.md](d:\Desktop\novel\docs\04_workflows\03_continue_workflow.md)
- [04_rewrite_workflow.md](d:\Desktop\novel\docs\04_workflows\04_rewrite_workflow.md)

### 选择理由

1. 自动小说系统最容易失控的不是“写不出来”，而是“越写越漂”
2. 没有审查入口，续写工作流就没有放行标准
3. 改写工作流也天然依赖审查结果和 `ReviewIssue`
4. 所以 `Review Workflow` 最适合作为 workflow 枢纽

### 影响范围

- 工作流层的展开顺序
- 审查与改写的闭环设计

### 风险 / 待验证项

- 先有审查不代表续写一定自然，后续仍需验证 `Continue Workflow` 是否真正受益于此顺序

### 后续动作

`keep`

当前先用单一 mock case 统一样例。

### 决策类型

`example`

### 决策状态

`adopted`

### 背景问题

进入示例层时，存在两个方向：

- 同时做多个题材样例
- 先用一个统一案例把整条链路跑通

### 备选方案

- 方案 A：多题材并行
- 方案 B：先统一使用一个 mock project

- `残雪归宗`

## 10. DEC-20260326-07

### 决策标题

当前优先验证连续性与审查闭环，不急于正文生成器。

### 决策类型

`project`

### 决策状态

`adopted`

### 背景问题

自动小说系统最诱人的方向，是尽快实现“自动出正文”。但这也最容易掩盖真正难的问题。

### 备选方案

- 方案 A：优先实现正文生成器
- 方案 B：先验证状态、连续性、审查与改写闭环

### 最终选择

当前先验证：

- 状态是否可维护
- 连续性是否可约束
- 审查是否能发现问题
- 改写是否能最小修复

暂不优先实现正文生成器。

### 选择理由

1. 没有连续性和审查，正文生成只会快速堆积坏结果
2. 当前项目的地基目标是“能稳定跑”，不是“先跑起来看起来像能用”
3. 先解决状态、事实、承诺、问题对象化，后面的正文生成才有真正落脚点

### 影响范围

- 当前项目目标
- 后续 experiment 的设计重点
- 与 Codex / 实现层的对接节奏

### 风险 / 待验证项

- 后续进入实现层时，需要避免因为文档地基过强而对正文生成策略形成不必要束缚

### 后续动作

`keep`

## 12. 共同特征

这 8 条决策有一个共同点：它们都不是为了让文档更完整，而是为了让系统更容易满足以下目标：

1. 边界清晰
2. 对象可运行
3. 规则可落地
4. workflow 可闭环
5. 后续实现不返工

这说明当前阶段的重点一直是稳地基，而不是堆功能。

---

## 13. 当前不建议轻易推翻的决策

以下决策建议视为当前地基级选择：

- 先文档地基，不先代码实现
- 核心对象固定为 8 个
- `FactLedger` 与 `ForeshadowGraph` 严格分层
- 规则层先于 workflow
- `Review Workflow` 先于 `Continue Workflow`
- 单一 mock case 优先

这些一旦被推翻，会牵动大量目录。
