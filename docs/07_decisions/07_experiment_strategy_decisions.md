# 实验推进策略决策

## 文档目的

本文用于收束当前阶段已经基本确定的实验推进策略决策。

它关注的不是某一条实验记录具体写了什么，而是更上层的推进问题：

- 实验应先验证什么，后验证什么
- 为什么当前先做结构验证，再做题材泛化验证
- mock case 连续试跑应如何分批推进
- 什么情况下可以认为某一层已经“基本成立”
- 什么情况下应从单案例验证转向多案例验证
- 实验层如何和决策层形成闭环

本文的作用，是防止后续实验推进时出现以下问题：

- 东做一个，西做一个，没有主线
- 一开始就做过重压力测试，导致根因混乱
- 结构还没站稳就急着做题材泛化
- 实验结论无法沉淀为设计修改

---

## 1. 文档定位

前面的决策文件已经固定了：

- 总方向
- 对象边界
- workflow 顺序
- schema 粒度
- 示例策略
- 审查策略

本文固定的是：

实验层如何从“有入口”推进到“有批次、有节奏、有验收标准”。

它主要影响：

- `06_experiments/`
- `07_decisions/`
- `05_examples/`
- 后续从 mock case 走向实验层的推进节奏

---

## 2. 当前需要固定的实验策略问题

当前最值得固定的实验策略问题主要有 8 个：

1. 为什么当前先做结构验证，不优先做题材泛化验证
2. 为什么实验应先跑单案例连续链路，而不是分散测试单点规则
3. 连续试跑应按什么批次推进
4. 什么叫“一层基本成立”
5. 什么时点从正例验证转向反例压力测试
6. 什么时点引入第二案例
7. 实验失败后如何进入决策修正
8. 当前最优先实验清单如何排序

## 3. DEC-20260326-48

### 决策标题

当前实验优先做结构验证，不优先做题材泛化验证

### 决策类型

`experiment`

### 决策状态

`adopted`

### 背景问题

当前系统已经完成了对象层、规则层、workflow 层、示例层和 mock case。下一步可以走两条不同方向：

- 结构验证：验证当前系统在单一案例上能否连续跑通
- 泛化验证：验证当前系统是否适用于不同题材、不同世界、不同叙事结构

### 备选方案

- 方案 A：现在就开始多题材泛化实验
- 方案 B：先把当前结构链路跑稳

### 最终选择

采用方案 B。

### 选择理由

1. 如果当前链路本身不稳，泛化实验只会放大噪声。
2. 当前更需要知道的是对象能不能长期维持、审查会不会漏判、改写会不会失控。
3. 只有结构验证先成立，后续泛化验证才有解释力。

### 影响范围

- `06_experiments/` 的前几批实验排序
- 第二案例的引入时机

### 风险 / 待验证项

- 如果首个案例选得过于“容易”，可能会高估当前结构稳定性
- 后续仍需补做题材泛化验证，不能把结构验证当作泛化成立

### 后续动作

`keep`

## 4. DEC-20260326-49

### 决策标题

实验应先跑单案例连续链路，而不是分散测试单点规则

### 决策类型

`experiment`

### 决策状态

`adopted`

### 背景问题

实验可以先做大量局部规则测试，例如单独测关系跳变识别、单独测时间线冲突识别、单独测伏笔遗失识别；也可以先跑一条最小连续链路，例如：

- `Continue`
- `Review`
- `Rewrite`
- `Continue`

### 备选方案

- 方案 A：先做大量单点规则实验
- 方案 B：先做单案例连续链路实验

### 最终选择

采用方案 B。

### 选择理由

1. 系统真正的风险不在单点规则，而在多层联动。
2. 连续链路最容易暴露状态未回写、warning 未升级、rewrite 只修表面不修对象等问题。
3. 单点规则实验仍然重要，但应该放在连续链路之后补做。

### 影响范围

- 首批实验文件的编排
- 实验优先级

### 风险 / 待验证项

- 如果连续链路样例过少，可能遗漏某些单点规则的边角失效

### 后续动作

`keep`

## 5. DEC-20260326-50

### 决策标题

连续试跑按“三批次”推进，而不是一次拉满复杂度

### 决策类型

`experiment`

### 决策状态

`adopted`

### 背景问题

mock case 的连续试跑可以一次性做得很复杂，也可以按复杂度逐批推进。

### 备选方案

- 方案 A：直接跑长链路、多线程、高复杂度实验
- 方案 B：按批次逐步提升复杂度

### 最终选择

采用方案 B。

### 选择理由

1. 分批更容易定位问题来源。
2. 一次拉满复杂度后，如果失败，很难判断问题来自 workflow 顺序、schema 粒度还是规则阈值。

### 影响范围

- `06_experiments/` 的批次规划

### 风险 / 待验证项

- 分批推进会拉长实验周期，需要避免因为节奏过慢而迟迟不进入压力测试

### 后续动作

`keep`

### 当前批次定义

#### 第一批：单轮闭环验证

目标：

- 从 `pu_scene_014` 出发，跑出一个 `pu_scene_015`
- 完成一次 `Continue`
- 完成一次 `Review`

#### 第二批：失败修复验证

目标：

- 人为制造一个错误续写版本
- 跑一次 `Review`
- 跑一次 `Rewrite`
- 再做一次复检

#### 第三批：多轮连续验证

目标：

- 连续跑 2 到 3 个 `PlotUnit`
- 观察 warning、issue、状态回写和承诺推进是否失控

## 6. DEC-20260326-51

### 决策标题

“一层基本成立”必须有最小通过标准，不能只凭感觉

### 决策类型

`experiment`

### 决策状态

`adopted`

### 背景问题

如果没有明确标准，就会不断陷入以下判断：

- 好像差不多了
- 应该能用了
- 看起来已经挺完整了

这种口径不利于决策层收束。

### 备选方案

- 方案 A：继续使用经验判断“基本成立”
- 方案 B：为各层设置最小通过标准

### 最终选择

采用方案 B。

### 选择理由

1. “基本成立”必须能被实验支撑，否则无法沉淀为后续决策。
2. 明确门槛后，实验层和决策层之间才有稳定接口。

### 影响范围

- 实验结论口径
- 决策日志的收束时机

### 风险 / 待验证项

- 当前标准仍偏“基础可用”，后续可能需要按对象层、规则层、workflow 层继续细化

### 后续动作

`patch`

### 当前最小通过标准

#### 对象层基本成立

满足：

1. 核心对象职责基本不冲突
2. mock case 能填通
3. workflow 使用时不会反复缺关键字段

#### 规则层基本成立

满足：

1. 正例能稳定解释成立原因
2. 反例能稳定抓到主要失败类型
3. 审查顺序能落地执行

#### workflow 层基本成立

满足：

1. Rebuild 能产出可运行状态
2. Continue 能产出有效 `PlotUnit`
3. Review 能决定分流
4. Rewrite 能最小修复一个明确问题

#### 示例层基本成立

满足：

1. mock case 能贯通主要对象
2. 至少一个强推进单元能被完整审查
3. 样例能被 workflow 复用

## 7. DEC-20260326-52

### 决策标题

反例压力测试应在第一批正向闭环之后立即补入，而不是无限后延

### 决策类型

`experiment`

### 决策状态

`adopted`

### 背景问题

虽然当前示例层采取“正例优先”，但反例如果拖得太久，会导致规则层验证偏乐观。

### 备选方案

- 方案 A：先长时间跑正例，再补反例
- 方案 B：正向闭环一旦成立，就立刻补第一批反例实验

### 最终选择

采用方案 B。

### 选择理由

1. 这能更早验证规则和 workflow 是否真的具备拦截与修复能力。
2. 第一批最常见失败类型最适合用来做压力入口。

### 影响范围

- `06_experiments/` 的第二批实验设计

### 风险 / 待验证项

- 第一批反例覆盖面仍有限，后续要根据实验结果决定是否继续扩展 failure type 集合

### 后续动作

`keep`

### 第一批建议反例

1. `relationship_jump`
2. `motivation_gap`
3. `missing_consequence`
4. `information_leak`

## 8. DEC-20260326-53

### 决策标题

引入第二案例的前提，是第一案例完成“连续两轮 + 一次修复闭环”

### 决策类型

`experiment`

### 决策状态

`adopted`

### 背景问题

第二案例何时引入，需要有明确门槛，否则很容易在第一案例尚未跑稳时过早扩展。

### 备选方案

- 方案 A：只要第一案例能跑一轮，就引入第二案例
- 方案 B：第一案例完成连续两轮并经历一次修复闭环后，再引入第二案例

### 最终选择

采用方案 B。

### 选择理由

1. 第二案例的目的，是验证泛化，而不是替代第一案例的结构验证。
2. 如果第一案例连“继续、出错、修复、再继续”都还没跑通，就还不具备扩展基础。

### 影响范围

- 示例扩展节奏
- 实验批次转段标准

### 风险 / 待验证项

- “连续两轮”是否已经足够暴露主要结构问题，仍需靠实验验证

### 后续动作

`watch`

### 当前门槛定义

满足以下条件后，才引入第二案例：

1. 第一案例连续成功推进至少两轮
2. 至少成功识别并修复一个正式问题
3. 修复后能重新放行继续推进
4. 未出现对象层大面积返工

## 9. DEC-20260326-54

### 决策标题

实验失败必须沉淀为“决策候选”，不能只停留在观察层

### 决策类型

`experiment`

### 决策状态

`adopted`

### 背景问题

实验日志如果只记录“发现了问题”，却不推动决策层，就会变成碎片堆积。

### 备选方案

- 方案 A：实验记录停留在 observation
- 方案 B：重要失败必须转成决策候选

### 最终选择

采用方案 B。

### 选择理由

1. 实验层负责发现问题。
2. 决策层负责固定边界和口径。
3. 两层不闭环，系统只会越试越散。

### 影响范围

- `06_experiments/`
- `07_decisions/`

### 风险 / 待验证项

- 需要避免把一次偶发失败过早升级成设计级决策

### 后续动作

`keep`

### 当前转化规则

当实验反复暴露以下情况时，应进入决策候选：

1. 同一边界问题重复出现
2. 同一 workflow 顺序问题重复出现
3. 同一字段经常不够用或明显冗余
4. 同一 warning 升级阈值反复模糊

## 10. DEC-20260326-55

### 决策标题

当前首批实验排序以“workflow 风险”优先，而不是“规则漂亮程度”优先

### 决策类型

`experiment`

### 决策状态

`adopted`

### 背景问题

实验优先级可以按不同逻辑排序：

- 哪个规则最重要
- 哪个 failure type 最核心
- 哪个 workflow 最容易失控

### 备选方案

- 方案 A：按规则重要性优先
- 方案 B：按 workflow 风险优先

### 最终选择

采用方案 B。

### 选择理由

1. workflow 失控会把对象、规则和审查一起拖垮。
2. 当前阶段最重要的，是让系统先“跑起来并站住”。
3. 规则细节可以在 workflow 基本稳定后继续打磨。

### 影响范围

- `06_experiments/00_experiment_log.md`
- 后续实验文件命名与顺序

### 风险 / 待验证项

- 如果 workflow 稳了但规则判定仍太粗，后续需要补足单点规则验证

### 后续动作

`keep`

### 当前首批实验推荐顺序

1. `Continue Workflow` 是否优先兑现近程后果
2. `Review Workflow` 能否识别关系跳变
3. `Rewrite Workflow` 能否最小修复一个动机问题
4. `FactLedger` 粒度是否稳定
5. `ForeshadowGraph` 多线程压力测试

## 11. 当前实验策略的总原则

基于以上决策，当前建议统一遵守以下原则：

1. 先验证链路，再验证泛化
2. 先跑单案例，再扩多案例
3. 先做正向闭环，再补第一批反例
4. 先跑低复杂度多轮，再上高复杂度压力
5. 实验必须推动决策，而不是只堆观察

## 12. 当前最值得继续验证的实验策略问题

虽然主策略已定，但以下问题仍需后续实验继续确认：

1. 连续两轮是否足够暴露系统主要问题
2. 第一批反例是否足够覆盖最常见失败
3. warning 升级阈值是否需要专门独立实验
4. 第二案例引入后，当前 workflow 是否会暴露题材偏置

## 14. 一句话总结

`07_experiment_strategy_decisions.md` 的作用，是把这套系统“实验先做什么、怎么分批、何时转段、怎样从实验进入决策”的推进策略固定下来，避免后面把实验层做成只有记录、没有主线、没有门槛、也没有收束点的试错堆。
