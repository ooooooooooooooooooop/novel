# 项目简介

## 项目名称
自动小说叙事系统

## 一句话目标
构建一个由作者与 AI 协同运行的小说解析、生成、审查系统，当前阶段先完成地基设计。

## 项目定位
这不是一个单纯的文本补全工具，而是一个面向小说创作的叙事状态系统。

系统最终希望支持：
- 解析小说结构
- 维护叙事状态
- 规划剧情推进
- 审查一致性与节奏问题
- 在后续阶段支持重建、续写、改写等能力

## 当前阶段
地基设计阶段

当前阶段重点：
- 定义核心概念
- 定义数据模型
- 定义状态转移规则
- 定义审查规则
- 建立最小工作流
- 准备样例与决策记录

## 为什么现在做
当前关于自动写小说的想法很多，但如果没有稳定的底层定义，后续实现会出现这些问题：
- 把文本误当成系统本体
- 角色与世界状态混乱
- 情节推进无法校验
- 审查标准不明确
- 系统目标不断膨胀

因此，需要先明确叙事系统的对象、边界、规则和判定方式。

## 当前项目核心判断
自动小说系统的第一公民不是正文，而是叙事状态。

小说应被理解为：
- 在规则约束下运行的虚构世界
- 由角色目标和冲突驱动的状态变化
- 被文本表达出来的叙事过程
- 可被记录、审查和修正的系统过程

### 3. 结构层文档

## 当前不追求
本阶段不追求：
- 技术实现
- 模型选型
- 基础设施部署
- 自动长篇完本
- 模拟某位具体作者
- 任意小说无缝续写
- 商业发行级文风
- 一步到位的产品化

## 第一阶段完成标准
当满足以下条件时，可认为地基阶段完成：

1. 项目目标和边界已明确
2. 8 个核心对象定义稳定
3. 主要字段结构已明确
4. PlotUnit 的状态转移逻辑已明确
5. 审查规则与失败类型已明确
6. 至少有一个 mock case 能串联整个系统定义

## 当前 checkpoint 判断

当前判断不是“地基还没搭起来”，而是：

- 地基骨架已经搭起来
- 三个高风险边界簇已有 first-pass convergence 锚点
- 当前最自然的动作是 phase judgment，而不是继续默认补局部反例

这同时意味着：

- 当前仍不建议进入实现阶段
- 当前更适合继续做 foundation checkpoint / transition judgment

## 当前 transition 判断

当前 transition 状态为：

- `ready_for_transition_planning`

原因不是设计缺块，而是：

- foundation gate 已通过
- 当前更适合基于既有边界锁进入 transition planning，而不是直接开实现

当前下一步为：

- 写清 transition planning 入口、允许讨论项、禁止重开项与 implementation-planning 前置产物

## 当前建议工作顺序

1. 项目定义
2. 核心概念
3. 数据模型
4. 状态转移规则
5. 审查规则

## 一句话总结
先把小说定义成一个可维护、可记录、可审查的叙事状态系统与 narrative agent harness，再考虑如何实现它。
